### 需求

### 方案确认

- `src/reporter/report_generator.py` — save_cross_validated/generate_cross_validated 新增 root_cause 参数，_build_cv_context 构建 rc_by_dim 映射+统计
- `src/reporter/templates/cross_validated_report.html` — 新增根因CSS(.rc-section/.rc-tag/.rc-insight等) + 问题项卡片内嵌入根因诊断子区域 + 根因总结框
- `PROJECT_STATUS.md` — v0.28 → v0.29

### 四种根因分类
1. 企业真实问题 — 公司基本面确实存在问题

## v0.30 结构化声明审计循环

### 需求背景
用户发现 v0.29 CV 三次跑茅台（v0.27/0.28/0.29）结果不稳定：31项核查→19项，问题项从5→4→1。根因分析：
1. CV 上下文截断（4000字 analysis + 12000 brief）导致每次核查片段不同
2. CV 无项目方法论背景知识，把 pipeline_calculation 误标为 ?缺乏证据
3. LLM 调用的固有漂移

### 方案设计
三管齐下：
1. **声明自拆 + 逐条核查**: 分析Agent提取原子声明 → CV逐条独立核查（每条≤2000字无需截断）
2. **CV专属方法论文档**: `rules/cv_methodology_brief.md` 注入 CV System Prompt（公式+打分逻辑），避免误判
3. **5种声明类型分策略核查**: pipeline_calculation(公式复算)/data_citation(对表)/trend_judgment(查Section三)/business_assertion(查Section五)/qualitative_score(不评判分数)

### 新增管线
```
Phase 5a: 分析Agent.analyze() → 完整报告
Phase 5a.5: 分析Agent.extract_claims() → 原子声明列表 (按5类型标注)
Phase 5b: CV.verify_claims_batch() → 逐条核查 VerifiedClaim[] (含cv_methodology_brief.md注入)
Phase 5b.5: 分析Agent.revise_with_cv_feedback() → RevisedClaim[] + 修正后报告
Phase 5c: 分析Agent.diagnose_root_cause() → 根因反思 (与v0.29相同)
```

### 新增文件 (2个)
- `rules/cv_methodology_brief.md` (~220行) — CV Agent 专属方法论文档：评分架构/PR公式/DC公式/L3十二维/OE标签/核查规则
- `src/llm/claim_types.py` (~200行) — AnalysisClaim/VerifiedClaim/RevisedClaim/ClaimVerificationResult 数据类

### 修改文件 (7个)
- `src/llm/analysis_agent.py` — +extract_claims()(提取原子声明,5种claim_type) + revise_with_cv_feedback()(accept/dispute/clarify三种回应)
- `src/llm/cross_validation_agent.py` — +_load_methodology()加载方法论文档 + _build_claim_check_system_prompt()注入系统提示 + verify_claim()单条核查(按类型分策略) + verify_claims_batch()批量核查 + _extract_brief_context()按声明类型截取相关Section
