## v0.24 — Web 搜索 + brief.md 数据底稿 + LLM 交叉验证

### 核心设计

用户设计了完整的新管线：
```
Phase 1: Tushare 数据快照（22接口）
Phase 2: L2-L5 量化打分 → FinalScore(100分制)
Phase 3: Web 搜索 → 广义商业研究（5类，DuckDuckGo）
Phase 4: 组装 brief.md（Tushare数据 + 管线得分 + Web搜索结果）
Phase 5: LLM 读 brief.md → 交叉验证（逐维对比，标注矛盾）
Phase 6: HTML 报告（含交叉验证结论）
```

关键设计决策：
- Web 搜索不是逐维定向搜索，而是 5 类广义商业研究（商业模式/管理层/行业地位/风险监管/分红回购）
- 搜索结果原文写入 brief.md，不做预筛选/预匹配
- LLM 自己读 brief.md 做逐维对比，搜索层不做定向映射
- brief.md 是数据底稿（三合一），作为 LLM 输入，不是输出品
- 搜索架构：`SearchBackend` 抽象 + `DuckDuckGoBackend` 默认（可插拔 Bing/SerpAPI）
- Web 搜索放在 Phase 3（打分之**后**），基于已得分结果进行验证

## v0.26 — 修复管线 vs 财报洞察 4/7 矛盾 Bug（Garbage-In-Garbage-Out）

### 问题根因
茅台交叉验证报告 7 个检查维度中 4 个矛盾（57%），根因是三条独立 Bug：
1. **财报深度分析取数粒度**：`_get_yearly()` 不区分季报/年报，Q1 营收 539亿 vs 年报 1688亿 直接做 CAGR → -24.9%
2. **DisposableCash 列名错误**：`st_borrow`/`tradable_fin_assets` 在 Tushare 中不存在（应为 `st_borr`/`trad_asset`），且 NaN 防御缺失
3. **L5 合理市值显示单位链**：从万元输出时除以 1e8 而非 1e4，导致 1.1万亿 显示为 1.1亿
